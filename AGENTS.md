# AGENTS.md — Agent 协作规则（严格执行版）

> 本文件规范所有 AI Agent 执行项目任务时的工作流程。
> **原则：任何涉及代码变更的任务，必须先 Explore 调研，再 Coder 修改。禁止在主会话直接写代码。**

---

## 0. 铁律（不可违反）

```
┌─────────────────────────────────────────────────────────────────┐
│  ❌ 禁止在主会话中直接修改任何代码文件（包括单行修改）              │
│  ❌ 禁止在主会话中直接运行任何测试（pytest、playwright、curl 等）   │
│  ❌ 禁止跳过 Explore Agent 直接启动 Coder Agent                   │
│  ❌ 禁止一个 Coder Agent 包办调研+修改+测试                         │
│  ✅ 所有任务必须：Explore 调研 → Coder 修改 → Coder 测试            │
└─────────────────────────────────────────────────────────────────┘
```

## 1. 强制使用子 Agent 的场景

以下任务类型**必须**通过子 Agent 执行。主会话只负责协调、决策、验收。

| 场景 | Agent 类型 | 职责 | 示例 |
|------|-----------|------|------|
| **代码库调研** | `explore` | 搜索文件、梳理架构、定位 Bug、分析依赖 | "sentiment 模块怎么工作的？" |
| **代码修改** | `coder` | 编写/修改代码、重构、修复 Bug | "给 TVL 增加 7d 变化列" |
| **测试验证** | `coder`（测试专项） | **代码审计** + **网络工具验证** + **Playwright 截图** | "测试 xxx 修改" |
| **安全审计** | `explore` | 检查权限、输入验证、敏感数据泄漏 | "检查回测端点权限控制" |
| **需求分析** | `plan` | 制定实现计划、识别关键文件、评估风险 | "设计新的智能路由模块" |

### 1.1 唯一例外：主会话可直接执行

以下操作**允许**在主会话直接完成，但**任何涉及文件写入/修改的都不行**：

- ✅ 读取已知路径的单个文件（只读，≤3 个文件）
- ✅ 回答纯知识性问题
- ❌ **修改任何代码文件**（即使是改一个字、删一行、加一个 import）
- ❌ **写入任何代码文件**（即使是创建空文件、写一行配置）
- ❌ **运行任何测试命令**（pytest、playwright、npm test、curl 验证等）
- ❌ **运行任何构建命令**（npm run build、vite build 等）

## 2. 标准工作流程（5 阶段 — 必须严格执行）

### Phase 1: 需求澄清
- 如果需求模糊或有多解，先用 `AskUserQuestion` 确认
- 将用户决策记录到 `memory/YYYY-MM-DD.md`

### Phase 2: 调研（Explore Agent）— **必须**
```
Agent(subagent_type="explore", prompt="...调研任务...", description="调研 xxx 模块")
```
- **必须提供完整上下文**：相关关键词、已知文件名、要回答的问题
- **调研深度**：
  - `quick`：找具体文件、函数、配置值
  - `medium`：理解模块工作原理、API 调用链
  - `thorough`：跨模块分析、数据流追踪、依赖映射
- **调研完成后，主会话基于结果制定实施计划**
- **何时可以跳过 Explore**：**永远不可以**。即使是改一行 import，也要先 Explore 确认这个 import 在哪里用、改了会不会影响其他地方。

### Phase 3: 实现（Coder Agent）— **必须**
```
Agent(subagent_type="coder", prompt="...修改任务...", description="修改 xxx")
```
- **必须提供**：文件路径、具体修改需求、约束条件（不要改测试、保持向后兼容等）
- **必须引用 Explore 的调研结果**
- Coder Agent 完成后返回：改了哪些文件、关键变更点、潜在风险
- 主会话 Review 代码变更，确认无误后进入 Phase 4

### Phase 4: 测试验证（测试 Agent）— **必须，主会话不得代劳**
```
Agent(
    subagent_type="coder",
    description="测试 xxx 修改",
    prompt="""
请对以下修改进行全面测试验证，必须包含三个维度：

**1. 代码审计**
- 检查修改是否引入了类型错误、未使用变量、空指针风险
- 检查是否有硬编码中文未走 i18n
- 检查是否有 console.log 遗留
- 运行 TypeScript 编译检查

**2. 网络工具验证**
- curl 验证相关 API 端点返回正确
- 检查响应时间是否符合预期
- 验证状态码和数据结构

**3. Playwright 截图验证**
- 运行 E2E 测试覆盖修改的页面
- 截图验证 UI 渲染正常
- 检查控制台是否有报错

**修改内容**: [简述]
**验证清单**:
- [ ] 代码审计通过
- [ ] API curl 验证通过
- [ ] Playwright E2E 通过
"""
)
```
- 修改完成后**必须**立即启动**测试 Agent**
- **主会话不得自行运行测试**（包括 pytest、playwright、curl、npm run build）
- **测试不通过 → 返回 Phase 3 修复**

### Phase 5: 验收与记录
- 验证截图/日志，确认符合预期
- 更新 `memory/YYYY-MM-DD.md` 记录变更摘要
- 如有长期影响，更新 `MEMORY.md`

---

## 3. 自检清单（每次任务启动前必须执行）

在动手之前，先在主会话中回答以下问题：

```
□ 这个任务是否需要修改任何文件？
  ├─ 是 → 必须启动 Explore Agent 调研
  └─ 否 → 可以在主会话直接执行

□ Explore Agent 是否已完成调研并返回结果？
  ├─ 是 → 基于调研结果启动 Coder Agent 修改
  └─ 否 → 不得启动 Coder Agent

□ 修改完成后是否启动了测试 Agent？
  ├─ 是 → 等待测试结果
  └─ 否 → 不得标记任务完成
```

---

## 4. 并发策略

- **独立任务可并发**：多个 explore agent 同时调研不同模块
- **依赖任务需串行**：调研 → 修改 → 测试，必须按顺序
- **善用 `run_in_background=true`**：E2E 测试、长耗时构建可后台运行，主会话继续其他工作

---

## 5. 禁止事项（红线 — 违反即违规）

| ❌ 禁止行为 | 正确做法 |
|-----------|---------|
| 在主会话直接修改任何代码（包括单行） | 启动 Coder Agent 执行 |
| 在主会话直接运行任何测试/构建 | 启动**测试 Agent** 执行（含代码审计+网络验证+截图） |
| 跳过 Explore 直接启动 Coder | 先 Explore 调研，后 Coder 修改 |
| 跳过测试直接标记完成 | Phase 4 测试 Agent 通过后才算完成 |
| 测试只跑单元测试不截图 | 测试 Agent 必须做代码审计+网络验证+Playwright 截图 |
| 不了解代码库就动手改 | 先启动 Explore Agent 调研 |
| 修改后不验证前端/API | 启动**测试 Agent** 截图/curl 验证 |
| 不记录关键决策 | 写入 memory/ 文件 |
| 一个 Agent 做所有事 | 按 Phase 拆分多个 Agent |

## 6. 子 Agent 启动模板

### 调研模板（每次任务必须先执行）
```
Agent(
    subagent_type="explore",
    description="调研 xxx 模块",
    prompt="""
请调研本项目的 xxx 模块，回答以下问题：
1. 相关文件在哪里？列出完整路径。
2. 核心类和函数有哪些？它们的职责是什么？
3. 与哪些其他模块有依赖关系？改了会不会影响其他地方？
4. 当前存在什么已知问题？
5. 如果要实现 [具体需求]，应该在哪些文件做修改？

调研深度：medium/thorough
"""
)
```

### 修改模板（必须基于 Explore 结果）
```
Agent(
    subagent_type="coder",
    description="修改 xxx 功能",
    prompt="""
请修改以下文件实现 xxx 功能：

**目标**: [具体描述]
**涉及文件**:
- path/to/file1.py
- path/to/file2.tsx

**约束**:
- 不要修改测试文件
- 保持现有代码风格
- 修改后运行相关测试验证

**参考上下文**:
[Explore Agent 的调研结果摘要]
"""
)
```

### 测试 Agent 模板（修改后必须执行）
```
Agent(
    subagent_type="coder",
    description="测试 xxx 修改",
    prompt="""
请对以下修改进行全面测试验证，必须覆盖三个维度：

**修改内容**: [简述]
**涉及文件**: [文件列表]

**1. 代码审计**
- 运行 `npx tsc --noEmit`（前端）或 `python3 -m py_compile`（后端）
- 检查是否有未使用变量、类型错误、空指针风险
- 检查是否有遗留的 console.log / debug 代码

**2. 网络工具验证**
- curl 验证 API 端点：`curl -s http://localhost:8000/api/xxx | python3 -m json.tool`
- 检查响应时间和状态码
- 验证数据结构是否符合预期

**3. Playwright 截图验证**
- 运行 E2E 测试：`npx playwright test tests/xxx.spec.ts`
- 截图验证修改的页面渲染正常
- 检查浏览器控制台是否有错误

**期望结果**: [具体预期]
"""
)
```

## 7. 记忆与知识沉淀

- **Daily Notes**: `memory/YYYY-MM-DD.md` — 当天所有任务的原始记录
- **Long-term**: `MEMORY.md` — 跨会话保留的关键决策、技术债务、架构约定
- **项目知识**: `docs/` — 设计文档、分析报告等可持久化知识
- **违规记录**: 本文件第 0 节「反面教材」— 记录违规事件作为警示
